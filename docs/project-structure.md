# Codebase & Project Structure

This document outlines the codebase layout, file naming conventions, module boundaries, and import rules.

---

## 📂 Core Folder Structure

The repository is divided into two main layers: `/backend` and `/frontend`.

```
Notepay_App/
├── .github/               # CI/CD workflow configuration files
├── backend/               # FastAPI backend codebase
│   ├── alembic/           # Alembic database migration scripts
│   ├── migrations/        # Autogenerated database migration history
│   ├── routers/           # Resource endpoints (admin, chat, events, transactions)
│   └── tests/             # Smoke and integration testing suites
├── docs/                  # Technical documentation and ADRs
│   ├── decisions/         # Architecture Decision Records
│   └── workflows/         # Feature visual workflow mappings
├── frontend/              # HTML, CSS, JS static assets
│   ├── assets/            # Fonts and image assets
│   ├── css/               # Modular styling sheets
│   └── js/                # Client-side JavaScript
│       └── controllers/   # Sub-view managers (Chat, Financials, Members)
├── serve_frontend.py      # Local python dev server (clean URL emulator)
└── template.yaml          # AWS SAM serverless infrastructure template
```

## 🔄 Module Communication & Boundaries

To prevent dependency cycles and maintain a clean separation of concerns, the codebase enforces the following boundaries:

```
[Frontend View] ---> [Page Controller] ---> [API client (api.js)]
                                                   |
                                            (HTTP Request)
                                                   |
[Database] <--- [CRUD Layer] <--- [Auth/Permissions Guard] <--- [FastAPI Router]
```

### 1. Frontend Boundaries
*   **Decoupled Pages**: Frontend views (e.g., `dashboard.html`) do not query backend databases directly. All API requests must route through the central API client `apiFetch()` in [api.js](../frontend/js/api.js).
*   **Shared Utilities**: Formatting and URL translation must be handled by `NPUtils` inside [shared-utils.js](../frontend/js/shared-utils.js) rather than writing custom logic inside views.

### 2. Backend Boundaries
*   **Router Responsibility**: Router files inside `/routers` are only responsible for parsing HTTP parameters, checking rate limits, and enforcing permissions. They must not contain business logic or database queries.
*   **CRUD Responsibility**: Database reads, inserts, updates, and deletes must reside in [crud.py](../backend/crud.py).
*   **Data Models**: Database schema structures are defined in [models.py](../backend/models.py). The schema models must be kept decoupled from the request/response validation logic defined in [schemas.py](../backend/schemas.py).

## 🏷️ Coding & File Naming Conventions

### 1. File Names
*   **Python Scripts**: Lowercase `snake_case` (e.g. `admin_auth.py`, `ws_manager.py`).
*   **JavaScript Scripts**: Hyphenated `kebab-case` for utility files (e.g. `auth-guard.js`, `shared-utils.js`) and `PascalCase` for sub-view controllers (e.g. `EventFinancialsController.js`).
*   **CSS Stylesheets**: Lowercase `kebab-case` (e.g. `global.css`, `dashboard.css`).
*   **HTML Templates**: Lowercase `kebab-case` (e.g. `create-event.html`, `join-event.html`).

### 2. Python Imports
*   Avoid circular imports by referencing parent objects or importing modules locally inside functions if needed.
*   Enforce absolute imports where possible:
    ```python
    # Correct
    from database import get_db
    import models
    
    # Incorrect
    from ..database import get_db
    ```

### 3. JavaScript Import Scopes
*   All utility files (`shared-utils.js`, `firebase-config.js`, `api.js`) must be loaded before page controllers in HTML scripts.
*   `NPUtils` is bound to the window object to provide global access to formatting helpers.
